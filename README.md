## What It Does

vigilex AI has four core features:

**1. Policy Analyzer**
Upload or paste any privacy policy (PDF or TXT). The system chunks the document, retrieves relevant clauses from a local legal database, and generates a structured compliance report — including a score, grade, compliance gaps, and missing elements — all grounded in real DPDP and GDPR law text.

**2. Legal Chatbot**
Ask plain-English questions about data protection law. The RAG pipeline retrieves the most relevant sections from the ingested legal PDFs and answers using those as context. Every answer cites its source documents.

**3. Compliance Posture Check**
An interactive checklist covering DPDP Act 2023, GDPR, and both frameworks. Tick what your organisation has implemented and get a weighted compliance score with a risk breakdown.

**4. Prompt Studio**
A minimal lower-corner prompt editor lets the team review, edit, save, and reset the live prompts used by Vigilex across chat, analyzer, and compliance flows without changing the top navigation.

    │  └── analyze_policy()      aggregate all findings → JSON report (Reduce)
    │
    ▼

    └── Embeddings: all-MiniLM-L6-v2 (CPU, HuggingFace)

           • it_act_2000_updated.pdf   (IT Act 2000)

### Map-Reduce Pipeline (Policy Analyzer)

```
Raw policy text
    │
    ▼  clean_policy_text()
Cleaned text
    │
    ▼  chunk_policy()  [1800 chars, 150 overlap]
[chunk1, chunk2, chunk3, ...]
    │
    ▼  MAP: for each chunk
    │    ChromaDB.similarity_search(chunk, k=2)
    │    → retrieve relevant law passages
    │    → LLM: identify strengths / gaps / missing elements
    │
    ▼  REDUCE
    Combine all chunk findings
    → LLM: synthesize into single JSON report
    │
    ▼
{score, grade, summary, strengths, gaps, missing_elements}
```

## Setup & Running

### Prerequisites

- Python 3.11+
- Node.js not required (plain ES modules, no bundler)
- ~1 GB free RAM (embedding model loads on first request)
- A GitHub Models API key ([get one here](https://github.com/marketplace/models))

### 1. Clone & install dependencies

```bash
git clone https://github.com/your-org/vigilex-ai.git
cd vigilex-ai

python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

### 2. Set your API key

Create a `.env` file in the project root:

```
GITHUB_INFERENCE_KEY=github_pat_your_key_here
```

> **Never commit `.env` to git.** It's already in `.gitignore`.

### 3. Build the vector database

Run once to index all PDFs in `data/` into ChromaDB:

```bash
python ingest.py
```

You should see `chroma_db/` populated. This only needs to be run again if you add new legal documents.

### 4. Start the backend server

```bash
python -m frontend.server   
```

Server starts at `http://localhost:5001`. You should see:

```
🚀 vigilex AI backend starting on http://localhost:5001
   POST /api/chat            RAG Q&A
   POST /api/upload-policy   PDF/TXT extraction
   POST /api/analyze-rag     Full compliance analysis
   POST /api/analyze         Direct LLM chat
   GET  /api/health          Sanity check
```

### 5. Open the app

Navigate to `http://localhost:5001` in your browser.

The Prompt Studio is available from the small `Prompts` gear button in the lower-right corner of the app.

### Example: analyze a policy

```bash
curl -X POST http://localhost:5001/api/analyze-rag \
  -H "Content-Type: application/json" \
  -d '{"policy_text": "We collect your name and email..."}'
```

Response:
```json
{
  "answer": "{\"score\": 42, \"grade\": \"Fair\", \"summary\": \"...\", \"strengths\": [...], \"gaps\": [...], \"missing_elements\": [...]}",
  "sources": ["dpdp-government.pdf", "GDPR.pdf"]
}
```

## Key Design Decisions

**Why local embeddings?**
`all-MiniLM-L6-v2` runs fully on CPU with ~380 MB RAM. No API costs for embeddings, no data leaving the machine during retrieval.

**Why map-reduce for policy analysis?**
A real privacy policy is 3,000–15,000 characters. Sending the full text in one prompt gives the LLM too much to reason about at once. Chunking into 1,800-character sections and doing targeted retrieval per chunk produces more specific, citation-grounded findings.

**Why lazy initialization?**
On an 8 GB machine, loading the embedding model + ChromaDB + Flask simultaneously at startup caused OOM kills. All heavy objects now initialize on first request, keeping startup memory flat.

**Why sequential map phase?**
Parallel chunk processing would spike RAM by N×chunk simultaneously. Sequential processing with explicit `gc.collect()` between chunks keeps peak RSS under 1 GB throughout analysis.
